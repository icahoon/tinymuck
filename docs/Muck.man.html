A New User's Introduction to TinyMUCK 2.3

Last updated [2/18/94] by Ken Arromdee. Converted to html by David Moore.

TinyMUCK 2.3 is derived from the original TinyMUD, which itself was inspired by the original MUD (Multi-User Dungeon). On a TinyMUD-derived mud, players can walk around, explore, program (in a Forth-derived stack-based language called MUF), and chat with other players. There's generally no goal to the game, unlike many other types of muds (LP Mud, DikuMUD) where players kill monsters and gain experience.

Connecting and Character Creation

To connect to a mud, use either telnet or one of the many client programs available. (A client program is a program you run in place of telnet to connect to a mud. This is uninformative; I know.) When you connect to the mud, you'll get a welcome screen. If you have a character, use the connect command; if you don't, use create. Some muds don't let new users create characters; in this case, you have to send electronic mail to the given address (if one is given), or connect with a guest character (which typically has a name of Guest and either no password or a password of guest).

Unlike IRC, you don't need a client to talk to the mud; all mud commands can be typed in directly, without any sort of protocol that has to be interpreted by your client. But clients can still help.

Basic Commands

The MUCK interface resembles a text adventure game. You can walk around in various directions by typing the direction, look at objects, pick up and drop things (using the get and drop commands), and check your inventory (which can be abbreviated down to i). There is no inventory size limitation. Players who build or program (see below) can define other commands; for instance, many MUCKs have commands like mail or news available MUCK-wide.

To talk to other players, you have four commands: say, pose, whisper, and page. The say command (abbreviated to " ) tells everyone in the room what you're saying, in quotes; "hi. or say hi. will give you the message You say, "hi." and (if your name is Guest) will tell others in the room Guest says, "hi.". The pose command (abbreviated to : ) will put the message directly after your name with no says; :waves. or pose waves. will tell everyone in the room Guest waves.. You usually just need the short forms " and : .

The whisper and page commands send messages to single players; the syntax is whisper player=message or page player=message. whisper only works on players in the same room. page works on any player on the MUCK; however, players can set their H flag to prevent being paged. (See below under flags.)

Many MUCKs have global commands which override the standard whisper and page; they provide extra features specific to the individual MUCK. Ask around on your MUCK to find out more.

The WHO command shows you the connected players and their connect times; it must be typed in capitals. You can find only players whose names start with a certain sequence of letters by typing WHO followed by the letters.

QUIT, which must be typed in all capitals, quits the game.

Abbreviation

Most built-in MUCK commands can be abbreviated; for instance, you can use w for whisper. Commands defined in the MUCK may or may not be abbreviated; in a well-built MUCK, they usually may.

Examining Objects

The command to examine an object is examine object. If you don't own the object, the command just tells you the owner. If you own the object, you get all the information about it: how many pennies it has, its flags, properties, and fields; its exits/actions, and any other useful information. This includes yourself; you may examine me.

Most of the commands mentioned below change an object in a way you can see with examine; if you don't know if the command worked, you can always examine the object and check.

Pennies

Pennies are a somewhat obsolete feature in MUCK that some MUCKs don't even have. If yours has them, you can find out how many pennies you have with the score command (or inventory, which also shows what you're carrying). You may sometimes find pennies randomly; MUF programs can also give you pennies. Building, and killing other players, uses pennies (killing uses up to 100).

You can give other players pennies with the command give player=amount. The typical penny limit is 10000. You can't give anything except pennies.

Matching

You can generally refer to objects by their name, or by some unique prefix. Sometimes you can use an object's number; that number is visible if you own it or if it has the A, C, or L flags (see below under Flags). For instance, if something showed up as magic hat(#1234L), and you wanted to pick it up, you could type get magic, get mag, get hat, get magic hat, or (if you own it) get #1234. If there was also something in the room named magic wand, you couldn't use get magic or get mag since the MUCK wouldn't know which one you mean; you'd have to use one of the unique forms. If there were two things named magic hat, you'd need to use the full name magic hat (which would pick a random one), or use the object number.

Although object numbers aren't always useful, it's important to know about them because some commands only work or only work well with numbers and not names.

The names me and here are special, and can be used anywhere normal object names can be. me refers to yourself, so look me would (for Guest) do the same as look Guest. here refers to the current room.

Going Home

Each player and thing has a home. When you type home, you and all of your things get sent to the appropriate home. @link me=room or @link thing=room will change your or the thing's home; the room must be specified with here or, if you're not in it, an object number, and the room must be owned by you or have the ABODE flag.

The starting home for players is one particular special room compiled into the game; for things, that's either the room where you created it (if you own the room or it has the A flag; see under flags), or your own home.

You can kill a player; this costs between 10-100 pennies for an equal percentage chance (100 for 100%). It gives them 50 pennies and sends them and their things home. It also prints a kill message; the drop field is shown to the person doing the killing, and the odrop field is shown to everyone else in the room. (See below under fields.) The format is kill player=cost. Killing doesn't work in rooms with an H flag, and wizards can't be killed.

Flags/Bits

Flags (also called bits) appear as letters after an object's number, if that's visible. For example, something displayed as rope(#5612S) has the S flag.

There are five types of objects on a MUCK: players, rooms, exits/actions, things and programs. All except things have a "flag" that shows what they are: P for players, R for rooms, E for exits/actions, and F for programs. You can't set these "flags".

The W flag on a player means that the player is a wizard; wizards maintain the database and run the mud, and can do almost anything. The M flag means the player is a MUCKER who is allowed to write MUF programs, and the B flag means the player is a BUILDER who is allowed to build and create objects. MUCKs can be set up so that all players may build or program, in which case the corresponding flags don't exist. You can't set these flags either (unless you're a wizard).

The other flags are A(BODE), C(HOWN_OK), D(ARK), H(AVEN), J(UMP)OK), L(INK_OK), Q(UELL), and S(TICKY). The command to set these is @set object=flag; @set me=A or @set me=ABODE will set the ABODE flag on yourself. To unset a flag, use a ! symbol; @set me=!A will remove the ABODE flag from yourself.

Not every flag works on every kind of objects. The Q flag is only good for wizards, and the A flag, when used on any object except a room, does nothing except make the object's number visible.

Properties

Properties are also set with the @set command; the syntax is @set object=property:value. To unset a property, @set object=property:, and to unset all properties, @set object=:.

Unlike flags, most properties are not used by built-in commands; lots of MUF programming (and some building) uses properties, but the exact properties which are useful change from MUCK to MUCK.

One exception is the sex property. The sex property is used for substitutions in the @ofail, @osuccess, and @odrop fields (see below, Fields). You should @set me=sex:male, @set me=sex:female, or @set me=sex:neuter so that those messages work properly.

Other exceptions are the connect and disconnect properties. These are shown as messages when you connect or disconnect from the mud.

Making Things

The command to create a thing is @create name. If you want to describe it so that people can look at it, you can @describe thing=message; otherwise, looking at it gives the message You see nothing special. (except for rooms, which are just blank). You can also describe yourself; one of the signs of a new player is that s/he hasn't yet given himself/herself a description.

If you set the S flag on a thing, it will go home when dropped.

To rename a thing (or any other object), @name object=newname. If you want to rename yourself, you must specify a password (separated by a space) after the new name.

Locks

To lock a thing to limit who can pick it up, use the @lock command: @lock thing=expression. The expression can use the following special characters: & (and), | (or), ! (not), and parentheses. You can refer to a player with a * before the name, and you can refer to a property with the name:value syntax. Refer to objects by name or number if they're in the same room, otherwise by number.

For instance, if boat and repellent were things, and #1000 were a room, the command 

> @lock squid=me | *SuicideSquid | swim:yes | (boat & !repellent & #1000)

would allow the following people to pick up the squid: you, the player named SuicideSquid, anyone with a property swim:yes on themselves or their inventory, and anyone who is carrying the boat thing, not carrying the repellent thing, and is in room 1000.

There is no special word you can use in a lock to specify that nobody can pick the thing up. Common ways to do this are to lock the thing to a room nobody can go to (such as #0 on many muds), or to lock to me&!me.

The command to remove a lock on something is @unlock thing.

Fields: succ, osucc, fail, ofail, drop, odrop

You can then set fields to print various messages; the commands to set all the different kinds fields are the same except for the name: @<field> thing=message. Don't confuse these with properties; for instance, to set an odrop field the command is @odrop thing=message, but not @set thing=odrop:message. (MUSH and MUSE do in fact make these forms equivalent, but MUCK does not.)

The succ and osucc fields are shown when someone picks up the thing. (If the thing is locked, this means they have to pass the lock.) succ is shown to the person picking it up. osucc is shown to the rest of the people in the room, but with pronoun substitutions and with the person's name added in front. Pronoun substitutions are as follows: 

Substitutions   sex:male  sex:female sex:neuter  (unset)
%a (absolute)       his        hers       its     <name>'s
%n         (%n is always replaced with the player's name)
%o (object)         him         her        it     <name>
%p (possessive)     his         her       its     <name>'s
%r (reflexive)  himself     herself    itself     <name>
%s (subject)         he         she        it     <name>

The substituted word is also capitalized if the letter after the % is capitalized.

For instance, if a thing had the osucc message reaches for the octopus. %S at first misses, but finally grabs the slimy creature in %p hands., and a male player named Guest picks it up, the room would see the message Guest reaches for the octopus. He at first misses, but finally grabs the slimy creature in his hands.

fail and ofail are the same, except are shown when you don't pass the lock.

drop and odrop are shown when someone drops the thing.

Rooms and Exits

The command to create a room is @dig name. This creates a room with that name, connected nowhere and with no entrances or exits, then prints the number of the new room. The command to make an exit is @open exit-name; the command to link that exit to a room is @link exit-name=room. In most cases, you can only specify the room by number, so if you're making rooms and exits and think the room number will scroll off your screen, you might want to write it down on paper. Neither of these commands automatically makes an exit back; to do that you have to go to the other room and make an exit separately.

The semicolon (;) is special in exit names, and separates alternate names for the same exit. If you wanted to be able to go somewhere by typing either w or west, you'd @open w;west.

The destination home is special as a destination; if you @link exit=home, anyone who uses the exit is sent home (but keeps everything s/he is carrying).

Exits and rooms may have descriptions; use the @desc command to describe them just like describing things.

Example (in this example, commands the player types start with > ): 

> look
Living Room(#1634R)
You are standing in a living room; a new MUDder has just moved
in, so it's still quite messy.  The floor is strewn with
cardboard boxes bearing indecipherable labels, and the only
walkable path between them leads eastwards to the kitchen.
> @dig Kitchen
Kitchen created with room number 1635.
> @open e;east;kitchen
Exit opened with number 1636.
> @link east=#1635
Linked to Kitchen(#1635R).
> east
Kitchen(#1635R)
> @describe here=This room is a kitchen east of the main living
room.  I'm going to give it a longer description later, since
bare descriptions look ugly.
Description set.
> look
Kitchen(#1635R)
This room is a kitchen east of the main living room.  I'm going to
give it a longer description later, since bare descriptions look
ugly.
> @open w;west;back;out;living;living room
Exit opened with number 1637.
> @link west=#1634
Linked to Living Room(#1634R).
> @describe west=You see a living room that way.
Description set.
> look west
You see a living room that way.

There is a shorthand form for opening and linking an exit, since they are commonly done together: @open name=room. In the above example, the first opening/linking pair could have been replaced by @open e;east;kitchen=#1635. If you do this, however, and make a mistake (such as typing the wrong room number), the opening might work while the linking fails. In this case, you don't want to use the shorthand form again; what that would do is open and link another exit, while leaving the first one present but unlinked. Instead, you want to separately type just the @link command.

You can only use the @open command in a room which you own. If you want an exit that comes from a room you don't own, you must ask the room owner to do the @open command, but not to link the exit; then you can go there and link the exit yourself. (You won't be able to use the shorthand form that combines opening and linking.)

You can only use the @link command to link an exit to a room you own, or to a room which has the L flag set. If you want an exit which goes to a room you don't own, you must ask the owner to set the room L.

The person who links the exit always ends up owning it.

Locks and Fields on Exits

The @lock command and all the field-setting commands work on exits. The lock controls who can go through the exit, so for instance if you wanted an exit named "ladies" to allow through either you, or females who carry a particular key, @lock ladies=me | (sex:female & key).

Setting a succ, osucc, fail, and ofail on the exit (with @succ, @osucc, @fail, @ofail) works as expected. If someone goes through the exit, s/he sees the succ message, and everyone else in the room sees the osucc message; the osucc message is pronoun substituted and the person's name added to the front. If the exit is locked so that someone can't go through it, s/he sees the fail message and the room sees the ofail message.

The fail and ofail messages are also shown if the exit is not linked anywhere. It's not a good idea, however, to leave an exit unlinked for this purpose, since anybody can relink it and steal the exit. If you want to make a scenery exit, link the exit to the same room (@link exit=here), and lock it to an unreachable room or an expression like me&!me, just like locking an ungettable object. You can use this to make fake things and to add commands to a room. In the example above: 

> @open box;boxes;cardboard;cardboard box;cardboard boxes=here
Exit opened with number 1638.
Linked to Living Room(#1634R).
> @lock boxes=me&!me
Locked.
> @desc boxes=The boxes all say "ACME Moving Company" on them.
Description set.
> @open open box;open boxes;tear box;tear boxes=here
Exit opened with number 1639.
Linked to Living Room(#1634R).
> @lock open boxes=me&!me
Locked.
> @fail open boxes=The boxes are tightly sealed.
Message set.
> @ofail open boxes=tries to open the boxes.

Now anyone in the room can look at the boxes and get the ACME description, and can type the command open boxes to get the appropriate messages.

Locks and Fields on Rooms

You might expect locks on rooms to keep out unwanted players. It won't work; you have to lock all the entrances into them (as well as unsetting the J, L, and A flags). You can lock, and set succ and other fields on, rooms; however, this affects only the room's description and messages.

If the room is locked, and someone passes the lock, they see the succ message as an extra line of description; otherwise they see the fail message. (The lock is checked whenever they'd see the description of the room: when entering and when typing the look command inside). The osucc (or ofail) messages are shown to everyone else in the room at the same time, with name and pronoun substitutions added.

The drop and odrop fields work differently on rooms; they are shown when someone drops a thing in the room. The odrop message is printed with the thing's name at the front, not the player's name.

You can set a drop-to on a room by @linking it to some other room, or to HOME. If somebody drops something there, the thing gets sent to the other room instead. If the room has an S flag, the things hang around until there are no people around, and then go to the other room; this feature is mostly a relic of TinyMUD.

Dark Rooms

If you set the D bit on a room, every object in the room becomes invisible except to you or to the object's owner. has arrived and has left messages don't appear.

Actions

Actions are exits, but don't have to come from, or go to, rooms. They are created with the command @action name=source. (Exits on rooms are a special case of actions, and the command @action name=here is the same as @open name.) The source for the action may be a room, a thing, or a person (i.e. you).

You can link an action to anything using the regular @link command; the object you are linking to must be owned by you or have its L flag set. The succ, fail, osucc, and ofail work properly on all actions, and you can still use semicolons in the action name to separate alternate names for the same action. Otherwise the effects are:

Rooms: Using the action takes you to the room (or your home, if the action was linked to the special `room' home), just like an exit. If the action is located on yourself, it's an easy way to go home without losing any of your inventory. If the action is on a thing, you have a portable instant transport to the room.

The drop and odrop fields work normally on actions that go to rooms; they are displayed in the destination room, the drop to the player and the odrop to everyone else (with name in front and substitutions).

Things: Using the action brings the thing to you. If the action's source is also located on a thing, that thing goes home. For example, consider a thing named glass of water with a drink water action, and a thing named empty glass with a fill glass action, each action linked to the opposite thing. Someone with the glass of water can type drink water and the glass will be replaced with the empty glass; when they type fill glass, the empty glass will be replaced with the full one.

If you don't want the source object to go home when the action retrieves the destination thing, you can set the S flag on the action.

The drop and odrop fields do nothing on actions which are linked to things.

Other Exits and Actions: Using an action linked to another action triggers the second action. For instance, you might have a glass with an action break glass, and want the action to cause the glass to vanish. To do that, you would make a hidden room with an action connected to the glass, then link the break glass action to this other action. Typing break glass would set off the room action, and the room action would bring the glass away from the player and into the room.

Players: Using an action that's linked to a player takes you to the room where the player is. The player must have his J flag set, or you get the message That player does not wish to be disturbed.. (Of course, he still has to be L for long enough for you to link the action to him in the first place.)

The drop and odrop fields on the action are displayed in the room the player is located in.

Programs: If you link an action to a program, using the action runs the program. See below under programs. The drop and odrop fields are unused.

Multiple Destinations: You can link an action to many other objects, if the extra objects are things or (other) actions. The syntax is @link action=object1;object2;object3;object4. For instance, if you typically carry around things coat(#2000) and hat(#2001), you can: 

> @action getstuff=me
Action created with number 2002 and attached.
> @link getstuff=coat;hat   (Or @link getstuff=#2000;#2001)
Linked to coat(#2000).
Linked to hat(#2001).

Then, every time you wanted to get your coat and hat, you could type getstuff.

Moving Actions

You can move actions/exits with the @attach command. @attach action=newsource moves the action to the new source. This doesn't change the destination; to change that you need to @unlink and @link again.

To move an action (or exit) from one room to another, attach it to yourself, go to the next room, and attach it to the room. This is cumbersome, but the @attach command doesn't allow you to specify actions that aren't in the room with you.

Actions and Room Parents

You can set a second room to be the parent of a room with the command @tel here=room2 (where the room is, of course, specified by number). The second room must be owned by you or have the L flag.

The purpose of a parent room is as follows: Any actions in the parent room (also called the environment room) work in the rooms inside it. For instance, if you had a house with many rooms, all with ceilings, and you wanted a message for up which said so, you could make one parent room. The parent room would have an up exit, locked, that just prints a message saying that there's a ceiling in the way. Then, you could parent all your house rooms to that one room. This would let people type up anywhere in your house and get the message.

Newly created start with one specific room as the parent, usually room #0. Extra commands that work throughout the MUCK (such as mail) are often implemented as actions that start in room #0 and are connected to MUF programs.

Action Matching

If there are several actions with the same name, actions are matched with the following priority: 

Actions on the room. (Highest priority.)
Actions on things you are carrying.
Actions on things in the room.
Actions on yourself.
Actions on parent rooms.  (Lowest priority.)

Actions can also override built-in commands, except for the built-in commands home, WHO, and QUIT. This may allow, for instance, custom page programs. If the action is named say or pose, it will override the normal commands with that name even if they are typed with the " or : abbreviations.

If several actions at the same level have the same name, the first one is selected with a 50% chance, and if not the next is, etc. until there's one left. For instance, if there are 4 actions/exits, there is a 1/2 chance of getting the first one, a 1/4 chance of the second, and a 1/8 chance of either of the two remaining exits.

Object Recycling

You can find out what things you own with the @find and @owned commands. @find lists everything except exits/actions, while @owned lists absolutely everything. You can further restrict @find with a parameter; @find dog would find objects whose names contain words starting with dog, like dog, hot dogs, or dogooder.

To get rid of something of yours, you @recycle it. The object is deleted from the database and it gets put on a list of garbage objects. (If it has exits or actions on it, the exits or actions get deleted too.) The next time someone makes something, one of the garbage objects will be reused. The garbage objects do use some database space, so if you create and recycle a thousand objects you don't get back all the space (unless there were a thousand garbage objects before you did the creation).

Changing Ownership

The @chown command allows you to change the ownership of something to yourself; the object must have the C flag on it. You also have to be carrying the object if it's an thing; this means that you can make a person a gift by setting the gift C and locking it to him/her. Only s/he can @chown it, since only s/he can pick it up.

Once you @chown something to yourself, you probably want to remove its C flag (@set thing=!C), and change its home if it's a thing, since its home is probably a room belonging to its old owner.

Running Programs

Writing and running MUF programs involves the @edit and @prog commands. (How to write programs is outside the scope of this document.) Most MUCKs are compiled so that you need to have an M flag set to use these commands; policies about who is allowed to have them differ widely. Ask around and read the news on your MUCK.

Using programs, however, doesn't require an M flag. There are three ways to use programs:

The first way is to use an action linked to the program. If an action is named b;bonk and linked to a bonk program, you can type bonk or b to run the program. MUCK saves any arguments, so if you typed b me or bonk me it would run the program with an argument of me. You could put this action on yourself if you wanted to be able to run it anywhere you went, or you could put it on an environment and be able to run it in any of the rooms in that environment.

The second way is to lock something to the program; the syntax is the same as locking to a thing. If you lock something to a thing, MUCK checks to see if the player is carrying the thing, but if you lock something to a program, MUCK runs the program when someone tries to manipulate the locked item. (Whether or not the person passes the lock depends on what the program does.) For instance, you could have a lock to a program which sends you a message; if you lock a thing to it, you would get the message whenever someone tries to pick it up, and if you locked an exit to it, you would get the message when someone goes through the exit. It's OK to have an exit with both a lock and a link to a program.

The third way is to put the program in a field; the acceptable fields for this are description, succ, fail, and drop. The syntax for giving an object a description that runs program #3129 with a parameter of fr00ty would be @desc thing=@3129 fr00ty. Whenever someone looks at the object, program 3129 would run with the given parameter.

The other fields work similarly: whenever they would be printed, the program runs instead. For instance, if program 1602 prints an explosion message, you could put @1602 in the drop field. Then, whenever someone dropped the object, the program would print the explosion message.

The exception is the succ, fail, and drop on people; programs in there don't work.

In order for you to run a program, either you must own it, it must have an L flag, or it must be on something owned by the program owner. (That is, the action linked to it, object locked to it, or object whose field contains it must be owned by the program owner).

Programs as Things

Programs themselves, as objects, generally act like things in other ways. They have descriptions, can be locked to keep people from picking them up, etc.... (Locking a program does not affect who may run it.) However, programs don't have homes (and don't go anywhere when you go home while carrying them), they are invisible to everyone except the owner unless they have an L flag, and you can't @chown them even if they have a C flag.

The DARK and STICKY flags have different meanings on programs; they become the DEBUG and SETUID flags.

The J Flag

The J (JUMP_OK) flag on something generally means that you can teleport it using a program. J things and programs can be teleported (unless they're in their owner's room or going to their owner's room, in which case the room must also be J); and J rooms can be teleported to by you. J people can be teleported if the rooms are also J.

The only non-program use of the J flag is the need for a J flag on a person for you to use an action that's linked to them. This is a double use of the J flag (known in computer science as overloading).

Creation with Cost

You can create a thing with a higher value with the command @create thing=cost. The cost is a number, and is taken from your pennies; the value of the thing becomes (cost-5)/5, with a maximum value of 100 for a cost of 505. This value (not the cost you paid!) is returned when you recycle the thing.

About all you can do with the value is check it with a MUF program.

Ages ago, when MUCK was TinyMUD, there was a T flag. Rooms with this were temples; when you dropped something there, it would go home and you would get pennies equal to the value. Puzzle solutions would provide prizes, that could be dropped in temples for pennies. Assuming that pennies still exist on your MUCK, the effect may be reproduced with a MUF program.

Custom % Substitutions

You can change the pronoun substitutions by setting properties named %a, %s, etc. on yourself; these will be used instead of the values derived from your sex property. This includes the %n property, so you can give yourself a longer name that will be used in some substitutions.

DARK flags

A person or thing with a DARK flag is invisible in a room's contents; a DARK person doesn't trigger the lock on the room or the room's succ, osucc, fail, or ofail fields, and gives no has arrived. or has left. messages. Exits with a DARK flag don't produce those messages either.

Only a wizard may set players, objects, or exits dark.

@entrances

The @entrances command finds everything with a link to something of yours. It's useful, but mostly for information; if you want someone to remove their link, you have to speak to them or to a wizard.

Prefixes and Suffixes

The commands OUTPUTPREFIX message and OUTPUTSUFFIX message provide the designated messages before and after any output you see. To cancel the message, use a null message. They cannot be abbreviated, cannot be overridden by actions, and must be typed in all capital letters. The commands are mostly useful to robots, not human players.

The Q Flag

The Q flag can be set on anyone, but only works for wizards; it cancels all special abilities of wizards with one exception: W programs owned by wizards still work.

Robbing

You can rob someone with rob person. This tries to steal one penny, and either does or doesn't, depending on whether they are locked; you can only steal pennies, and only one at a time. The person's succ, osucc, fail, and ofail fields are used for this purpose. Programs in the succ and fail fields don't run, though if the person is locked to a program that program will run.

It is therefore a good idea to immediately lock yourself so that nobody can rob you, when you get a new character on a mud: @lock me=me. The rob command is a relic of TinyMUD, and is not (nor was) really ever useful for stealing.

Passwords

The command to set your password is @password new=old. The command @newpassword is not the same thing.

Teleporting

The @teleport command (syntax: @teleport thing=destination) can be used to teleport things around; however, the destination must either have an A flag or be owned by you. The thing must be yours or in a room you own. You can teleport things to yourself.

The @teleport command is also used for setting room parents.

A reasonably written teleport program is generally more useful than the @teleport command.

@version

The @version command does the obvious.
